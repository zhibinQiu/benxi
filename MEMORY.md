## 📝 其他需要记住的内容

- **代码分析范围**：分析代码时只分析前端（frontend/）和后端（backend/）目录，不需要分析所有文件（如 config/、docs/ 等）。
- **倒计时提醒方式**：提醒类任务（"8s 后提醒我喝水"），用 `schedule_notification` 工具，参数名 **`scheduled_at`**，传 ISO 8601 绝对时间（当前时间+延迟）。
- **系统 MEMORY.md**：平台 AI 记忆文件通过对象存储 `agent-memory/{user_id}/MEMORY.md` 管理，通过平台前端「AI 设置 → 智能体记忆」编辑。本 Cursor 工作区 MEMORY.md 仅用于 IDE AI 助手，两者独立。
- **数据持久化**：项目数据目录在服务器 `/root/qzb/benxi/data`，容器内通过 `DATA_ROOT` 映射（compose.yaml 中 `${DATA_ROOT:-../../data}`）。数据包含 PostgreSQL、MinIO、pdf2zh 配置等，非常重要，需定期备份。
  - 备份方式：服务器上 `bash scripts/stack.sh backup`（备份到 `backups/` 目录）
  - 同步代码时附带备份：`./dev.sh sync --with-data`（在服务器端执行备份，不拉到本地）
  - 仅备份：`./dev.sh sync --backup`（不同步代码，只做服务器端备份）
  - 定时备份脚本：`scripts/server-backup.sh`（可通过 cron 安装，默认保留 30 天）
- **品牌名**：项目名称统一为「本析」（benxi），没有任何 `lvye` 相关的目录或配置。

## 📝 代码注释与文档规范

- **只写现状，不写过程**：代码注释和文档只描述当前是怎么实现的、原理是什么，不写改了什么、之前哪错了。
  - ✅ 好的注释：`// 使用双缓冲队列避免频繁加锁，reader 读取 bufA 时 writer 写入 bufB`
  - ❌ 差的注释：`// 这里之前写错了，改成双缓冲方式，之前的写法有并发问题`
  - 如果历史上下文确实重要（如兼容旧数据的迁移逻辑），可以在注释中简要说明约束条件，但不要把注释写成"改版日志"。

## 🏗️ 系统设计原则

- **无必要勿增实体**：不做过度设计，不做无用的抽象，不做不存在的功能。
- **能复用尽量复用**：提取公共方法，消除重复代码，避免到处粘贴。
- **保持服务稳定性与独立性**：服务之间松耦合，不因一个模块的改动影响其他模块。
- **前端轻量化**：
  - 优先用 Naive UI 等现成组件，减少不必要的自定义封装，关键复用逻辑才抽组件，防止多层样式覆盖
  - 组件化拆分，提取公共逻辑到 composable，避免在页面中写大量重复逻辑
  - 减少资源占用（按需加载、懒加载、及时销毁监听与定时器）
  - 加快响应速度，不做不必要的渲染和计算
- **后端抽象化**：
  - 逻辑分层清晰，抽离公共能力到 service / domain 层，避免在 API 路由中堆砌业务逻辑
  - 简化分支，减少响应时间，避免深层嵌套和多余 IO
  - 提高鲁棒性：关键路径加熔断、重试、超时控制，做好错误分类处理
  - 提高并发性：合理使用异步/协程、连接池，避免串行化瓶颈
- **系统角色设定优先级**：构建智能体 system prompt 时，各层设定按以下优先级叠加（高→低）：
  1. **当前对话的设定** — 运行时注入的 `context_instruction`、路由原因、临时指令
  2. **Skill** — skill catalog、playbook 指令
  3. **Style** — STYLE.md 回复风格
  4. **Memory** — 用户记忆（`build_memory_prompt_context`）
  5. **系统默认的设定** — 默认 AGENT.md、resident prompt 常驻块

## 🌐 语言策略

- **中间环节一律用英文**：智能体之间的交互消息、注入 LLM 的 system prompt/tool 描述/上下文块、错误消息等，均使用英文。
- **最终输出保留用户语言**：用户输入中文则最终回答为中文，用户输入英文则最终回答为英文。—— 本规则已记录，下次对话起自动遵循。
- **本析平台 Agent 回复中不出现「ROUTE」**：本析平台的 AI Agent 在给平台用户的最终回复文本中，不得包含「ROUTE」这个词（包括其大小写变体），操作结果应通过自然语言描述而非标注「ROUTE」来体现。
- **Agent 禁止编造执行结果**：调用工具、Skill 或子智能体后必须拿到真实结果才回复用户；调用失败时也必须如实告知错误信息，禁止为掩盖失败而编造数据。**调用失败后不得主动提供与用户原始请求无关的替代服务**（如技能无法执行时说"我帮您搜索类似信息"）。失败即报告，让用户决定下一步。
- **Agent 禁止假装执行**：不得在未实际调用 tool 的情况下写"正在搜索…""请稍等…"等空话。只有实际调用了工具后才能描述执行进度。

## 💾 保存系统版本的工作流

当我说「保存系统版本」时，请执行以下操作：

1. **代码清理** — 扫描并移除前后端失效的、错误的、冗余的代码，包括但不限于：
   - 未引用的变量、函数、import
   - 废弃的 API 路由和视图
   - 死注释和空壳文件
   - 冗余的条件分支和重复逻辑
2. **文档同步** — 更新相关文档，确保其反映当前代码状态
3. **Git 版本更新** — 更新 `VERSION` 文件中的版本号
4. **Bug 根治** — 若发现 bug，找出根本原因并修复，禁止打补丁式的临时修复
