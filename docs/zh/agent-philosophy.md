# 智能体系统设计哲学

> 本文档阐述本析平台智能体系统的设计理念、角色定义与交互方式。

## 为什么需要多种智能体？

单一智能体在面对复杂任务时存在以下问题：

- **工具膨胀**：将所有工具塞给一个 LLM，prompt 过长导致注意力分散、决策退化
- **上下文污染**：规划步骤与执行细节混杂，工具结果互相干扰
- **领域耦合**：文档库操作、浏览器 RPA、联网搜索的 Schema 完全不同，混在一起增加幻觉
- **效率低下**：简单查询也要加载全套工具、走完整规划流程

我们的解决方案是 **分层解耦 + 扁平交互**——将能力拆解为四个层级的实体，层间通过轻量协议通信。

## 四层实体模型

```

用户提问
    │
    ▼
┌─────────────────────────────────────┐
│  ① 调度智能体（Orchestrator）         │
│  原子工具直答 / 子智能体 / 路由专精    │
└──────────┬──────────────────────────┘
           │
    ┌──────┼──────────┬──────────────┐
    ▼      ▼          ▼              ▼
 原子工具  子智能体   专精智能体      Skill 脚本
 (searxng) (deep_     (platform/     (上传包)
            research)  rpa/skill-dev)
```

### 第①层：调度智能体（Orchestrator）— 「小析」

**定位**：用户的第一接触点，统一入口。

**职责**：
- 接收用户所有消息
- 用原子工具快速处理简单问题（直接搜索、查知识库、发通知等）
- 判断任务复杂度，按路由规则分配给下层实体
- 汇总各层返回结果，整理成面向用户的最终回复

**为什么需要调度层？**
- 80% 的用户请求是简单查询（天气、时间、名称搜索），无需复杂路由
- 避免用户每次等待完整的「规划 → 路由 → 执行 → 汇总」流程
- 调度层是用户与系统的"缓冲区"，能答就答，不能答再分发

### 第②层：子智能体（Sub-agent）— 「研究员」

**定位**：需要动脑子但不涉及专业领域的任务。

**模式**：
| 模式 | 用途 | 说明 |
|------|------|------|
| `deep_research` | 深度联网研究 | 多轮搜索、交叉验证、结构化报告 |
| `explore` | 多源并行检索 | 同时搜网络/知识库/图谱，快速摘要 |
| `browser_digest` | 页面内容取证 | 打开指定 URL 获取页面结构与内容 |

**为什么要子智能体？**
- 调研类任务需要多轮、多角度、自主规划——不适合作为「工具」暴露
- 子智能体拥有独立的 LLM 上下文和 tool loop，可以自主思考
- 与其让调度器反复调用搜索工具，不如让子智能体专注研究
- 研究和对话分离：调度器不做研究，子智能体不做回复——各司其职

**交互方式**：调度器通过 `invoke_context_subagent(kind=xxx, task=...)` 发起，子智能体完成任务并返回结构化结果。

### 第③层：专精智能体（Specialist）— 「专家」

**定位**：特定业务领域的执行专家。

**现有专精**：
| 专精ID | 职责 |
|--------|------|
| `platform` | 平台文档库 CRUD、待办管理、用户/部门管理 |
| `rpa` | 浏览器自动化操作（导航/点击/填表/截图） |
| `skill-dev` | 上传型 Skill 生命周期管理 |
| `report` | 结构化报告撰写（可研/需求/方案/调研/测试/工作计划） |

**为什么需要专精？**
- 每个专精只加载自己领域的 Skill 和工具，prompt 精简，减少幻觉
- 专精之间的工具互不干扰（rpa 不会看到文档库的 API）
- 专精可以独立迭代、独立挂载前端配置

**交互方式**：
- 调度器通过 `request_orchestrator_assist(suggested_agent_id=xxx)` 路由任务
- 专精在自己的 tool loop 中完成操作
- 完成后将结果返回调度器，由调度器汇总回复

### 第④层：Skill 脚本 — 「工具人」

**定位**：不需要动脑子、直接执行的傻瓜式操作。

**形式**：上传型 Python 脚本（main.py），系统提供标准运行时环境。

**适用场景**：
- 定时抓取网站数据
- 格式化输出报告
- 对接外部 API

**为什么需要 Skill？**
- 用户自定义能力扩展，无需修改平台代码
- 与核心系统隔离，脚本异常不影响主服务

## 层级间如何交互？

### 1. 调度 → 原子工具（第①层内）

调度器直接调用工具函数，无需间接层：

```
web_search / knowledge_retrieve / kg_query / fetch_url_content
send_notification / schedule_notification
read_agent_memory / append_agent_memory
ask_user_choice
```

### 2. 调度 → 子智能体（第①→②层）

通过 `invoke_context_subagent(kind, task)` 发起：

```
调度器                       子智能体
  │                            │
  ├─ invoke_context_subagent ─►│
  │                            ├─ web_search (多轮)
  │                            ├─ knowledge_retrieve
  │                            ├─ 分析 & 交叉验证
  │                            ├─ 返回结构化结果
  │◄─ 返回研究报告 ───────────┤
  │                            │
  ├─ 基于结果写回复            │
  │                            │
```

### 3. 调度 → 专精（第①→③层）

通过 `request_orchestrator_assist(suggested_agent_id)` 发起：

```
调度器                       专精
  │                            │
  ├─ request_orchestrator_assist ─►│
  │                            ├─ invoke_skill (领域 Skill)
  │                            ├─ 执行操作
  │◄─ 返回结果 ───────────────┤
  │                            │
  ├─ 汇总回复                  │
  │                            │
```

### 4. 专精 ←→ 调度（第③→①层）

专精遇到瓶颈时，通过 `request_orchestrator_assist()` 向调度器求助：

```
专精                                 调度器
  │                                    │
  ├─ 发现自己无法完成                   │
  ├─ request_orchestrator_assist() ──►  │
  │                                    ├─ 协调另一专精
  │                                    ├─ 或其他方式
  │◄─ 协助结果 ──────────────────────┤
  │                                    │
  ├─ 继续完成本域任务                   │
```

## 决策原则

1. **能简则简**：第①层能答的，绝不走到第②层
2. **扁平优先**：子智能体能做的，不路由到专精
3. **业务隔离**：文档操作、浏览器 RPA 必须走专精，不在调度层混入领域工具
4. **结果导向**：只收结论，不收探索过程
5. **拒绝否定**：遇到阻碍时说明原因并给出可操作建议，不简单说"做不到"

## 为什么不用全路由式架构？

全路由（所有请求先进 Planner 再路由）的问题：
- 简单问题也要排队走规划流程，延迟增加
- Planner 跟调度器的决策边界模糊，容易互相矛盾
- 层层转发，调试困难

我们的扁平化设计：
- 调度器先用原子工具试答，试不动再路由
- 子智能体专注于"思考"和"研究"，不做决策
- 专精专注于"执行"，不负责规划
- 每一层都是独立的 LLM 上下文，互不污染
