# 平台权限模型说明

本文说明 **账号权限（RBAC）** 与 **文档权限（分级 + ACL）** 如何分工，以及数据安全上的推荐用法。

## 两套系统各管什么

| 层级 | 作用 | 典型配置 |
|------|------|----------|
| **账号 / 功能权限** | 能否登录、能否进入某功能、能否在「公司/部门/个人」库**新建**文档 | 角色：`sys_admin`、`company_admin`、`dept_admin`、`member`；权限码如 `doc.read`、`doc.dept.edit` |
| **文档分级（scope）** | 文档默认落在哪一级库、默认谁能看见 | `personal` / `department` / `company`；文档库另有 **「所有」** Tab，汇总可查询及以上权限的文档 |
| **单文档 ACL** | 在默认规则之外的**例外** | 显式 `document_permissions`；`document_access_denials` 禁止 |

原则：**默认靠分级 + 角色；单文档授权只用于例外**（例如个人文档分享给同事），不要靠复制文档或全员 admin。

## 单文档显式授权四档（`document_permissions.level`）

| 级别 | 代码 | 能力 |
|------|------|------|
| 可见 | `visible` | 列表、详情、下载、预览 |
| 可查询 | `query` | 可见 + 知识问答 / KnowFlow 检索、**PDF 翻译**、文档对比 |
| 可编辑 | `edit` | 可查询 + 上传新版本（**不含删除**） |
| 完全 | `full` | 可编辑 + 删除文档（通常仅上传者与分级管理员） |

兼容旧值：`read`→`visible`，`use`→`edit`，`delete`→`full`（启动时自动迁移）。

`GET /documents/{id}/access-control` 返回 `can_view` / `can_query` / `can_edit` / `can_delete` / `effective_level`。

## 默认可见规则（无显式授权时）

1. **个人级**：仅创建人可见、可用。
2. **部门级**：同部门且具备 `doc.read` 的成员默认可见；创建人始终可见。
3. **公司级**：具备 `doc.read` 的用户默认可见。

判断顺序（读权限）：

1. 系统管理员 → 允许  
2. 存在显式授权（`document_permissions`，级别 ≥ visible）→ 允许  
3. 在 `document_access_denials` 中 → **拒绝**（覆盖部门/公司默认可见）  
4. 按 scope + 角色默认规则  

KnowFlow 侧通过分级知识库 + `ragflow_kb_acl` 同步，与平台 ACL 一致；**部门库只存一份文档**，多人通过 RBAC 授权，不复制文件。

## 谁能做什么（关键规则）

### 单文档「授权」（grant / revoke）

- **仅文档创建人** 与 **系统管理员**（`sys_admin` / `admin.user`）。
- **即时生效，无需被授权方审核**。
- 部门/公司管理员 **不能** 给单文档添加显式授权（避免越权扩散）。

### 「禁止访问」（deny）

- 创建人、系统管理员：任意文档均可操作。
- **部门管理员**：本部门 **部门级** 文档，可禁止本部门某成员访问（收回默认可见）。
- **公司管理员**：**公司级** 文档，可禁止某公司成员访问。

## 文档状态与回收站

- **启用 / 关闭**：`active` / `disabled`；关闭后普通用户不可见，上传者与分级管理员仍可管理。
- **删除**：移入**操作者个人回收站**（`deleted_by`），不立即物理删除文件；可在「回收站」Tab 点**恢复**。
- **管理权限**（`can_manage`）：上传者、系统管理员、公司/部门管理员（对应分级 `doc.*.edit`）；用于改状态、删除、恢复。

## 实用性建议

1. **个人协作**：创建人在文档详情用 **用户列表多选** 授权（可见/可查询/可编辑/完全），勿手填 UUID。  
2. **部门/公司管控**：优先用 **禁止访问** 屏蔽个别员工，而不是给每人单独 grant。  
3. **公司/部门库**：由公司/部门管理员直接在公司级或部门级新建文档；成员默认可见、可检索。  
4. **权限码 `doc.grant`**：不再用于单文档授权；保留给历史兼容时可从角色中移除，避免与「仅创建人授权」混淆。  
5. **审计**：生产环境建议记录 grant/deny 操作日志（当前以数据库记录 + `granted_by` 为主）。

## API 摘要

| 接口 | 说明 |
|------|------|
| `GET /documents/{id}/access-control` | 返回 `can_grant` / `can_deny` / `is_owner` 及四档能力位 |
| `GET /documents/{id}/acl-candidates` | 可选用户列表（授权/禁止弹窗） |
| `POST /documents/{id}/permissions` | 显式授权（创建人或系统管理员） |
| `POST /documents/{id}/denials` | 禁止访问（见上） |
