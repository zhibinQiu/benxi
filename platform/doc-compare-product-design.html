    文档对比功能 · 产品设计（评审稿 v0.1）

    供评估后进入开发。与平台插件 doc_compare、feature.doc_compare 及 KnowFlow 检索/标注能力对齐。

    1. 产品定位

    在企业文档库中，对 2–4 份 用户已有权限的文档进行：

    1. 差异对比（Diff）：增删改以荧光笔底色标注，支持侧栏差异列表与预览联动定位。
    2. 自然语言定位（Search）：用一句话在多篇文档中检索相关段落并高亮跳转。

    设计原则：文档权限以平台 DocumentPermission + can_access_document 为唯一真相；KnowFlow 负责解析、分块、语义召回与（可选）坐标；对外仅暴露平台 API。

    2. 用户与场景

    场景 说明
    合同/协议版本对照 初稿、修订稿、终稿 2–3 版
    制度文件年度修订 条款增删改
    招投标文件对齐 招标要求 vs 响应书
    多译稿/多语言措辞差异 文本层对比为主

    首期非目标：协同编辑、像素级图 diff、复杂 Legal 红线导出（可后续 Phase）。

    3. 双模式设计

    模式 A：差异对比（主路径）

    flowchart LR
  Select[选择 2-4 文档] --> Parse[解析文本层]
  Parse --> Diff[平台 diff 引擎]
  Diff --> UI[多栏预览 + 荧光高亮]
  UI --> List[侧栏差异列表]
    • 输入：从文档库多选（doc.use），指定基准文档（默认第一份）。
    • 处理：抽取可对齐文本（PDF 文本层或 KnowFlow/MinerU 解析结果）。
    • 输出：段落/句子级 diff；类型：新增、删除、修改、移动。
    • 交互：底色高亮 + 点击侧栏条目 → 各栏同步滚动定位。

    模式 B：自然语言定位（辅助 / Phase 3）

    • 输入：自然语言问句（如「违约金」「数据出境义务」）。
    • 范围：当前任务已选的 2–4 个 document_id。
    • 处理：平台计算 ACL 白名单 → KnowFlow 检索（metadata platform_document_id in (...)）→ 返回 chunk + 页码/bbox。
    • 输出：命中列表；点击后在对应栏高亮（紫色系，与 diff 色系区分）。

    4. 页面与信息架构

    路由建议：/system/compare（系统功能入口）、/system/compare?job={id}（恢复任务）。

    步骤 页面区块
    1 选文档：多选 2–4；仅展示已上传且用户 doc.use 的 PDF/Word 等
    2 配置：基准文档、对比粒度（段落/句子）、是否开启 NL
    3 工作台：顶部 Tab + NL 搜索框；中部 2–4 列预览；右侧差异/检索抽屉
    4 结果：任务保存、导出差异摘要（Markdown/CSV）

    工作台布局（桌面）

    ┌─────────────────────────────────────────────────────────────┐
│ [Doc A*] [Doc B] [Doc C]   🔍 自然语言搜索…    [导出][保存] │
├───────────────────────────────────┬─────────────────────────┤
│  预览 A（高亮） │ 预览 B │ 预览 C  │  差异列表 │ NL 命中      │
│               │        │         │  + 12     │  3 条       │
│               │        │         │  ~ 5      │             │
│               │        │         │  − 2      │             │
└───────────────────────────────────┴─────────────────────────┘

    5. 高亮与标注规范

    类型 底色 侧栏 说明
    删除 浅红 − 相对基准缺失内容
    新增 浅绿 + 对比稿新增
    修改 浅黄 ~ 左右原文对照
    移动 浅蓝 ↕ 段落重排
    NL 命中 浅紫 ⌕ 检索定位，可与 diff 叠加

    需提供：仅显示差异、按文档筛选、hover 显示页码/偏移。

    6. 权限模型

    检查点 规则
    功能入口 feature.doc_compare
    选文档 / 抽取 每文档 doc.use
    预览 doc.read
    对比任务 created_by = 当前用户；列表仅本人（管理员策略可扩展）
    NL 检索 先算 allowed_document_ids，再调 KnowFlow

    与 PDF 翻译、文档库列表逻辑一致，不使用 KnowFlow 自带 RBAC 做文档级授权。

    7. KnowFlow 集成要点

    环节 做法
    解析 对比前触发 parse/index（或复用已有索引）；metadata 含 platform_document_id, version_id
    NL 检索 检索 API + metadata 过滤；返回 snippet、score、page、bbox
    坐标高亮 Phase 4：PDF.js 上叠加 bbox（依赖 KnowFlow MinerU 坐标）
    删除文档 平台删文档时同步删 KnowFlow 文档/chunk

    对接前需确认：检索是否支持 metadata 过滤；是否返回 bbox；更新版本策略。

    8. 数据与 API（草案）

    表：compare_jobs、compare_diff_items、compare_search_hits；可选 document_index_refs（与 RAG 共用）。

    API（前缀 /api/v1/compare）：

    • GET /documents — 可对比文档列表
    • POST /jobs — 创建任务
    • GET /jobs/{id} — 进度与 diff 列表
    • POST /jobs/{id}/search — NL 检索
    • GET /jobs/{id}/export — 导出摘要

    实现形态：Feature 插件 doc_compare，Celery 异步 parse+diff，SSE 进度（可参考 pdf_translate）。

    9. 分阶段交付建议

    阶段 范围 周期（估）
    MVP 双文档、文本 diff、荧光高亮、侧栏列表、权限 2–3 周
    P2 2–4 文档、基准模式、多栏联动 +1–2 周
    P3 NL 跨文档定位（KnowFlow） +2 周（依赖 API）
    P4 PDF 坐标级高亮、导出带标注 PDF 视 KnowFlow 能力

    10. 评审问题清单

    1. 主场景是版本演进还是平行文档？是否默认「基准文档」？
    2. 2–4 份是否必须同类型（如都是合同）？
    3. NL 搜索是 MVP 必选项还是 P3？
    4. 是否需要导出带高亮 PDF？
    5. KnowFlow 环境版本与 metadata 检索 API 是否已就绪？
    6. 对比粒度默认段落还是句子？

    11. 与现有平台的关系

    • 插件注册：app/features/builtin/doc_compare.py（当前为 stub）
    • 文档来源：MinIO + document_versions（与翻译从文档库选 PDF 相同）
    • 任务模型：可复用 Job 类型 doc_compare 或独立 CompareJob 表

    确认本设计后，可进入技术方案细化与编码。
